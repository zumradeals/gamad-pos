# **ROADMAP V1 — IKOMA POS**

## **De la gestion du commerce local à GAMAD Business Network**

### **Vision de long terme**

**Rendre visibles, organisés et interconnectés les commerces, produits, stocks, fournisseurs et besoins économiques africains.**

IKOMA POS constitue le premier point d’entrée opérationnel.  
GAMAD Business Network deviendra progressivement le réseau économique continental.  
GAMAD Core sera le socle fédérateur de l’identité, des organisations, des autorisations et des services partagés.

---

# **1\. Architecture stratégique générale**

La vision repose sur trois niveaux strictement séparés.

## **Niveau 1 — IKOMA POS**

Application opérationnelle privée utilisée quotidiennement par les commerces.

Elle gère :

* ventes ;  
* produits ;  
* stocks ;  
* clients ;  
* créances ;  
* encaissements ;  
* livraisons ;  
* dépenses ;  
* achats ;  
* fournisseurs ;  
* équipes ;  
* points de vente ;  
* clôtures ;  
* abonnements.

IKOMA POS doit rester :

* simple ;  
* mobile ;  
* accessible aux non-techniciens ;  
* utilisable avec une faible connexion ;  
* compréhensible par des utilisateurs peu alphabétisés.

## **Niveau 2 — IKOMA Business Network**

Couche de mise en réseau économique locale.

Elle permet :

* publication volontaire de catalogues ;  
* découverte locale des produits ;  
* recherche du meilleur prix ;  
* demandes de devis ;  
* recherche de fournisseurs ;  
* achats groupés ;  
* mise en relation entre commerçants ;  
* visibilité des stocks dormants ;  
* échanges entre points de vente ;  
* recherche de produits rares ;  
* réputation commerciale.

## **Niveau 3 — GAMAD Business Network**

Extension continentale et intersectorielle du réseau IKOMA.

Elle reliera progressivement :

* commerçants ;  
* artisans ;  
* producteurs ;  
* importateurs ;  
* grossistes ;  
* distributeurs ;  
* transporteurs ;  
* prestataires ;  
* clients particuliers ;  
* entreprises ;  
* institutions.

## **Socle transversal — GAMAD Core**

GAMAD Core ne doit pas être une copie du POS. Il doit devenir le socle commun chargé de :

* l’identité numérique ;  
* l’authentification ;  
* les organisations ;  
* les rôles ;  
* les autorisations ;  
* les abonnements ;  
* les événements interservices ;  
* les référentiels partagés ;  
* la gouvernance ;  
* la traçabilité ;  
* les connexions avec les futurs modules GAMAD.

# **2\. Architecture de marque recommandée**

Il ne faut pas renommer prématurément IKOMA POS. Il faut organiser une famille de produits.

## **Marque opérationnelle**

### **IKOMA POS**

Vendre, gérer, suivre.

Produit quotidien du commerçant.

## **Extensions possibles**

* **IKOMA Market** : recherche publique de produits ;  
* **IKOMA Supplier** : fournisseurs et grossistes ;  
* **IKOMA Connect** : demandes et mises en relation ;  
* **IKOMA Delivery** : livraisons et logistique ;  
* **IKOMA Shop** : équipements compatibles ;  
* **IKOMA Insights** : analyses commerciales ;  
* **IKOMA Academy** : formation des commerçants ;  
* **IKOMA Assist** : assistance et accompagnement ;  
* **IKOMA Catalogue** : publication et synchronisation des catalogues.

## **Évolution institutionnelle**

Quand le réseau dépassera IKOMA et couvrira plusieurs secteurs et pays :

### **GAMAD Business Network**

IKOMA POS restera alors une application spécialisée connectée au réseau, et non un ancien nom à supprimer.

# **3\. Principes fondateurs à verrouiller**

Ces principes doivent guider toutes les versions.

## **3.1 Le commerce reste propriétaire de ses données**

IKOMA ne doit jamais publier automatiquement :

* les stocks exacts ;  
* les coûts d’achat ;  
* les marges ;  
* les clients ;  
* les créances ;  
* les fournisseurs privés ;  
* les performances des employés.

## **3.2 La publication est volontaire**

Chaque commerce choisit :

* ce qui reste privé ;  
* ce qui est partagé avec ses partenaires ;  
* ce qui est rendu public ;  
* ce qui est indexable par le réseau.

## **3.3 Le paiement intégré aux ventes n’est pas obligatoire**

IKOMA POS enregistre :

* espèces ;  
* Wave ;  
* Orange Money ;  
* MTN Money ;  
* Moov Money ;  
* virement ;  
* paiement différé.

Mais le commerçant continue d’utiliser ses propres comptes.

## **3.4 Genius Pay sert principalement aux abonnements IKOMA**

Genius Pay intervient pour :

* souscrire ;  
* renouveler ;  
* recevoir une facture ;  
* vérifier l’abonnement ;  
* réactiver un compte ;  
* gérer les échéances.

## **3.5 Le fournisseur de paiement reste remplaçable**

Le domaine métier doit parler d’« abonnement » et de « paiement d’abonnement », jamais directement de Genius Pay dans sa logique centrale.

## **3.6 Le POS reste simple**

Les futurs modules ne doivent pas transformer l’écran du vendeur en ERP.

Chaque utilisateur voit uniquement ce qui correspond à son rôle.

# **4\. Les personas structurants**

La conception doit reposer sur des profils distincts.

## **Propriétaire**

Il veut savoir :

* combien il a vendu ;  
* combien il a encaissé ;  
* qui lui doit de l’argent ;  
* quels produits manquent ;  
* ce que font ses vendeurs ;  
* combien il gagne réellement.

## **Vendeur**

Il veut :

* vendre rapidement ;  
* retrouver un produit ;  
* choisir une quantité ;  
* encaisser ;  
* enregistrer une dette ;  
* indiquer si le client emporte maintenant.

## **Caissier**

Il gère :

* encaissements ;  
* reçus ;  
* caisse ;  
* entrées et sorties ;  
* clôture.

## **Magasinier**

Il gère :

* réception ;  
* sorties ;  
* transferts ;  
* inventaire ;  
* corrections autorisées ;  
* préparation des commandes.

## **Livreur**

Il voit :

* commandes à livrer ;  
* client ;  
* téléphone ;  
* lieu ;  
* marchandises ;  
* montant à récupérer ;  
* preuve de livraison.

## **Fournisseur**

Il souhaite :

* publier son catalogue ;  
* recevoir des demandes ;  
* trouver des revendeurs ;  
* annoncer ses prix ;  
* suivre ses commandes.

## **Client**

Il souhaite :

* trouver un produit ;  
* comparer les vendeurs ;  
* vérifier la disponibilité ;  
* contacter une boutique ;  
* demander un devis ;  
* acheter près de lui.

## **Administrateur IKOMA**

Il gère :

* abonnements ;  
* entreprises ;  
* assistance ;  
* incidents ;  
* modules ;  
* statistiques d’usage ;  
* conformité ;  
* modération.

# **9\. Phase 4 — Catalogue découvrable**

## **Objectif**

Faire passer IKOMA d’un outil privé à une infrastructure commerciale volontairement ouverte.

## **9.1 Niveaux de visibilité**

Chaque produit peut être :

### **Privé**

Visible uniquement par l’entreprise.

### **Partenaire**

Visible par des entreprises autorisées.

### **Réseau professionnel**

Visible par les membres IKOMA ou GAMAD.

### **Public**

Visible par tous les utilisateurs du moteur de recherche.

## **9.2 Options de publication**

Le commerçant choisit de publier :

* le nom ;  
* la photo ;  
* la description ;  
* le prix ;  
* une fourchette ;  
* la disponibilité ;  
* la quantité exacte ;  
* une quantité approximative ;  
* la zone ;  
* le contact ;  
* les modalités de livraison.

## **9.3 Protection commerciale**

Toujours masquer par défaut :

* prix d’achat ;  
* marge ;  
* stock privé ;  
* fournisseurs ;  
* ventes ;  
* clients ;  
* historique.

## **9.4 Fraîcheur de l’information**

Chaque résultat doit indiquer :

* mis à jour aujourd’hui ;  
* mis à jour récemment ;  
* disponibilité à confirmer ;  
* information ancienne.

Le système ne doit jamais présenter un stock ancien comme une certitude.

### **Répertoire**

### **Événements**

# **21\. Contrats de cohérence incontournables**

Toute modification de stock génère un mouvement traçable.

Une vente validée ne peut pas être supprimée silencieusement.

Une créance exige un débiteur identifiable.

Une livraison différée exige un destinataire retrouvable.

Une remise exceptionnelle exige un motif et éventuellement une autorisation.

Le catalogue privé et le catalogue public sont séparés.

La publication d’un produit est un consentement explicite.

La disponibilité publiée possède une date de fraîcheur.

L’abonnement IKOMA est indépendant du fournisseur de paiement.

GAMAD Core ne doit pas contenir la logique métier propre au POS.

Les modules communiquent par des contrats explicites.
